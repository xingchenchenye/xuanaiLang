# 玄言 / XuanLang

玄言是一门正在成长中的中文编程语言。

这个项目不是在做“中文关键字套壳”，也不是只摆一份语法草图出来讲故事。它现在已经能真正运行代码，也有自己的编译前端、解释执行、Python 后端和 VM 字节码后端。换句话说，玄言已经过了空想阶段，正在往一门认真可用的语言原型走。

开发者署名：`开发者星尘_尘夜`

## 这是什么

玄言想解决的是一个很具体的问题：

如果一门语言从一开始就把中文表达、结构化数据、编译链演进和 AI 推理场景放在一起设计，它会长成什么样？

所以在这个仓库里，你会看到几件事同时出现：

- 中文语义为主，英文符号辅助。
- 完整的编译前端，而不是只有语法高亮式演示。
- 三条可运行的执行链，方便后续继续往 AOT、JIT 和硬件后端扩展。
- 提前接入张量、KV Cache、attention、transformer step 这类 AI 运行时原型。

## 现在做到哪一步了

如果要很实在地说，玄言目前还不是成熟语言，也还远没有到“生产可替代”的程度。但它已经是一套可以拿出来跑、查、改、测、继续长的工程了。

当前版本已经具备：

- 词法分析、语法分析、AST、语义分析、优化器、IR
- 解释器执行链
- Python 编译后端
- 独立 VM 字节码执行与反汇编
- 函数、条件、循环、匹配、模块导入
- 静态类型标注与基础类型推导
- 列表、映射、结构体
- 成员赋值、索引赋值、复杂左值赋值
- 张量、矩阵乘法、量化入口、KV Cache、attention、transformer step 原型
- 项目脚手架、格式化、检查、测试、示例程序

仓库内当前附带 `30` 项自动化测试，用来校验解释器、Python 后端、VM 后端、项目工具和主要语法能力。

## 它现在适合做什么

如果你现在把玄言拉下来，比较适合拿它做这些事：

- 看一门中文语言原型是怎么一步步长出来的
- 做编译器、解释器、虚拟机方向的实验
- 试验中文语义下的结构化表达
- 验证面向 AI 推理场景的运行时设计
- 在这个底座上继续往更强的类型系统、执行后端和硬件适配推进

如果你期待的是一门现成、成熟、生态完整的通用语言，那玄言现在还不是那个阶段。

## 一个实际例子

下面这段代码已经可以直接运行：

```xuan
构 引擎 {
    名称: 文
    版本: 整
    配置: 映[文, 文]
}

术 升级(目标: 引擎) -> 整 {
    目标.版本 = 目标.版本 + 1；
    目标.配置["后端"] = "vm"；
    返 目标.版本；
}

令 核心: 引擎 = 新 引擎 {
    名称: "玄言核",
    版本: 1,
    配置: {"后端": "解释器"}
}；

显("升级前:", 核心)；
令 版本 = 升级(核心)；
显("升级后:", 核心, 版本)；
```

这段例子背后已经覆盖了结构体、映射、函数、类型标注、成员赋值、索引赋值和标准输出。

## 当前能力一览

| 模块 | 状态 | 说明 |
| --- | --- | --- |
| 语言前端 | 已实现 | Lexer、Parser、AST、Semantic、Optimizer、IR |
| 解释执行 | 已实现 | 可直接运行玄言源码 |
| Python 后端 | 已实现 | 可编译为 Python 执行 |
| VM 后端 | 已实现 | 有独立字节码执行与反汇编能力 |
| 类型系统 | 初步可用 | 支持标注、推导、映射和结构体类型 |
| 数据结构 | 初步可用 | 列表、映射、结构体 |
| AI 运行时 | 原型可用 | 张量、量化入口、KV Cache、attention、transformer step |
| 工程工具 | 已实现 | 新建项目、运行、检查、格式化、测试 |
| 生态系统 | 早期阶段 | 还没有包管理器、调试器、编辑器插件 |
| 原生硬件后端 | 未实现 | AOT / JIT / CUDA / ROCm / NPU 仍在路线图中 |

## 当前边界

玄言现在最准确的定位是：一门已经跑通核心链路、并且明显朝 AI 原生方向生长的早期语言内核。

它还没有完成这些事：

- 真正的 AOT / JIT 双引擎
- 原生机器码后端
- 全平台硬件后端适配
- 完整的 C / C++ / Python 双向 ABI
- 包管理器、调试器、编辑器插件生态
- 更大规模、更稳定的标准库体系

所以如果你是来找一个已经“什么都齐了”的语言，它还不是；但如果你想看一个方向明确、底座已经搭起来、而且还在快速演进的语言项目，玄言现在已经值得关注。
